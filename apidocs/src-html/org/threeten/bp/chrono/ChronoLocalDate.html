049/**
050 * A date without time-of-day or time-zone in an arbitrary chronology, intended
051 * for advanced globalization use cases.
052 * <p>
053 * <b>Most applications should declare method signatures, fields and variables
054 * as {@link LocalDate}, not this interface.</b>
055 * <p>
056 * A {@code ChronoLocalDate} is the abstract representation of a date where the
057 * {@code Chrono chronology}, or calendar system, is pluggable.
058 * The date is defined in terms of fields expressed by {@link TemporalField},
059 * where most common implementations are defined in {@link ChronoField}.
060 * The chronology defines how the calendar system operates and the meaning of
061 * the standard fields.
062 *
063 * <h4>When to use this interface</h4>
064 * The design of the API encourages the use of {@code LocalDate} rather than this
065 * interface, even in the case where the application needs to deal with multiple
066 * calendar systems. The rationale for this is explored in the following documentation.
067 * <p>
068 * The primary use case where this interface should be used is where the generic
069 * type parameter {@code <C>} is fully defined as a specific chronology.
070 * In that case, the assumptions of that chronology are known at development
071 * time and specified in the code.
072 * <p>
073 * When the chronology is defined in the generic type parameter as ? or otherwise
074 * unknown at development time, the rest of the discussion below applies.
075 * <p>
076 * To emphasize the point, declaring a method signature, field or variable as this
077 * interface type can initially seem like the sensible way to globalize an application,
078 * however it is usually the wrong approach.
079 * As such, it should be considered an application-wide architectural decision to choose
080 * to use this interface as opposed to {@code LocalDate}.
081 *
082 * <h4>Architectural issues to consider</h4>
083 * These are some of the points that must be considered before using this interface
084 * throughout an application.
085 * <p>
086 * 1) Applications using this interface, as opposed to using just {@code LocalDate},
087 * face a significantly higher probability of bugs. This is because the calendar system
088 * in use is not known at development time. A key cause of bugs is where the developer
089 * applies assumptions from their day-to-day knowledge of the ISO calendar system
090 * to code that is intended to deal with any arbitrary calendar system.
091 * The section below outlines how those assumptions can cause problems
092 * The primary mechanism for reducing this increased risk of bugs is a strong code review process.
093 * This should also be considered a extra cost in maintenance for the lifetime of the code.
094 * <p>
095 * 2) This interface does not enforce immutability of implementations.
096 * While the implementation notes indicate that all implementations must be immutable
097 * there is nothing in the code or type system to enforce this. Any method declared
098 * to accept a {@code ChronoLocalDate} could therefore be passed a poorly or
099 * maliciously written mutable implementation.
100 * <p>
101 * 3) Applications using this interface  must consider the impact of eras.
102 * {@code LocalDate} shields users from the concept of eras, by ensuring that {@code getYear()}
103 * returns the proleptic year. That decision ensures that developers can think of
104 * {@code LocalDate} instances as consisting of three fields - year, month-of-year and day-of-month.
105 * By contrast, users of this interface must think of dates as consisting of four fields -
106 * era, year-of-era, month-of-year and day-of-month. The extra era field is frequently
107 * forgotten, yet it is of vital importance to dates in an arbitrary calendar system.
108 * For example, in the Japanese calendar system, the era represents the reign of an Emperor.
109 * Whenever one reign ends and another starts, the year-of-era is reset to one.
110 * <p>
111 * 4) The only agreed international standard for passing a date between two systems
112 * is the ISO-8601 standard which requires the ISO calendar system. Using this interface
113 * throughout the application will inevitably lead to the requirement to pass the date
114 * across a network or component boundary, requiring an application specific protocol or format.
115 * <p>
116 * 5) Long term persistence, such as a database, will almost always only accept dates in the
117 * ISO-8601 calendar system (or the related Julian-Gregorian). Passing around dates in other
118 * calendar systems increases the complications of interacting with persistence.
119 * <p>
120 * 6) Most of the time, passing a {@code ChronoLocalDate} throughout an application
121 * is unnecessary, as discussed in the last section below.
122 *
123 * <h4>False assumptions causing bugs in multi-calendar system code</h4>
124 * As indicated above, there are many issues to consider when try to use and manipulate a
125 * date in an arbitrary calendar system. These are some of the key issues.
126 * <p>
127 * Code that queries the day-of-month and assumes that the value will never be more than
128 * 31 is invalid. Some calendar systems have more than 31 days in some months.
129 * <p>
130 * Code that adds 12 months to a date and assumes that a year has been added is invalid.
131 * Some calendar systems have a different number of months, such as 13 in the Coptic or Ethiopic.
132 * <p>
133 * Code that adds one month to a date and assumes that the month-of-year value will increase
134 * by one or wrap to the next year is invalid. Some calendar systems have a variable number
135 * of months in a year, such as the Hebrew.
136 * <p>
137 * Code that adds one month, then adds a second one month and assumes that the day-of-month
138 * will remain close to its original value is invalid. Some calendar systems have a large difference
139 * between the length of the longest month and the length of the shortest month.
140 * For example, the Coptic or Ethiopic have 12 months of 30 days and 1 month of 5 days.
141 * <p>
142 * Code that adds seven days and assumes that a week has been added is invalid.
143 * Some calendar systems have weeks of other than seven days, such as the French Revolutionary.
144 * <p>
145 * Code that assumes that because the year of {@code date1} is greater than the year of {@code date2}
146 * then {@code date1} is after {@code date2} is invalid. This is invalid for all calendar systems
147 * when referring to the year-of-era, and especially untrue of the Japanese calendar system
148 * where the year-of-era restarts with the reign of every new Emperor.
149 * <p>
150 * Code that treats month-of-year one and day-of-month one as the start of the year is invalid.
151 * Not all calendar systems start the year when the month value is one.
152 * <p>
153 * In general, manipulating a date, and even querying a date, is wide open to bugs when the
154 * calendar system is unknown at development time. This is why it is essential that code using
155 * this interface is subjected to additional code reviews. It is also why an architectural
156 * decision to avoid this interface type is usually the correct one.
157 *
158 * <h4>Using LocalDate instead</h4>
159 * The primary alternative to using this interface throughout your application is as follows.
160 * <p><ul>
161 * <li>Declare all method signatures referring to dates in terms of {@code LocalDate}.
162 * <li>Either store the chronology (calendar system) in the user profile or lookup
163 *  the chronology from the user locale
164 * <li>Convert the ISO {@code LocalDate} to and from the user's preferred calendar system during
165 *  printing and parsing
166 * </ul><p>
167 * This approach treats the problem of globalized calendar systems as a localization issue
168 * and confines it to the UI layer. This approach is in keeping with other localization
169 * issues in the java platform.
170 * <p>
171 * As discussed above, performing calculations on a date where the rules of the calendar system
172 * are pluggable requires skill and is not recommended.
173 * Fortunately, the need to perform calculations on a date in an arbitrary calendar system
174 * is extremely rare. For example, it is highly unlikely that the business rules of a library
175 * book rental scheme will allow rentals to be for one month, where meaning of the month
176 * is dependent on the user's preferred calendar system.
177 * <p>
178 * A key use case for calculations on a date in an arbitrary calendar system is producing
179 * a month-by-month calendar for display and user interaction. Again, this is a UI issue,
180 * and use of this interface solely within a few methods of the UI layer may be justified.
181 * <p>
182 * In any other part of the system, where a date must be manipulated in a calendar system
183 * other than ISO, the use case will generally specify the calendar system to use.
184 * For example, an application may need to calculate the next Islamic or Hebrew holiday
185 * which may require manipulating the date.
186 * This kind of use case can be handled as follows:
187 * <p><ul>
188 * <li>start from the ISO {@code LocalDate} being passed to the method
189 * <li>convert the date to the alternate calendar system, which for this use case is known
190 *  rather than arbitrary
191 * <li>perform the calculation
192 * <li>convert back to {@code LocalDate}
193 * </ul><p>
194 * Developers writing low-level frameworks or libraries should also avoid this interface.
195 * Instead, one of the two general purpose access interfaces should be used.
196 * Use {@link TemporalAccessor} if read-only access is required, or use {@link Temporal}
197 * if read-write access is required.
198 *
199 * <h3>Specification for implementors</h3>
200 * This interface must be implemented with care to ensure other classes operate correctly.
201 * All implementations that can be instantiated must be final, immutable and thread-safe.
202 * Subclasses should be Serializable wherever possible.
203 * <p>
204 * Additional calendar systems may be added to the system.
205 * See {@link Chronology} for more details.
206 *
207 * @param <D> the date type
208 */
209public interface ChronoLocalDate<D extends ChronoLocalDate<D>>
210        extends Temporal, TemporalAdjuster, Comparable<ChronoLocalDate<?>> {
211
212    /**
213     * Comparator for two {@code ChronoLocalDate}s ignoring the chronology.
214     * <p>
215     * This comparator differs from the comparison in {@link #compareTo} in that it
216     * only compares the underlying date and not the chronology.
217     * This allows dates in different calendar systems to be compared based
218     * on the time-line position.
219     * This is equivalent to using {@code Long.compare(date1.toEpochDay(),  date2.toEpochDay())}.
220     *
221     * @see #isAfter
222     * @see #isBefore
223     * @see #isEqual
224     */
225    public static final Comparator<ChronoLocalDate<?>> DATE_COMPARATOR =
226            new Comparator<ChronoLocalDate<?>>() {
227        @Override
228        public int compare(ChronoLocalDate<?> date1, ChronoLocalDate<?> date2) {
229            return Long.compare(date1.toEpochDay(), date2.toEpochDay());
230        }
231    };
232
233    /**
234     * Gets the chronology of this date.
235     * <p>
236     * The {@code Chronology} represents the calendar system in use.
237     * The era and other fields in {@link ChronoField} are defined by the chronology.
238     *
239     * @return the chronology, not null
240     */
241    Chronology getChronology();
242
243    /**
244     * Gets the era, as defined by the chronology.
245     * <p>
246     * The era is, conceptually, the largest division of the time-line.
247     * Most calendar systems have a single epoch dividing the time-line into two eras.
248     * However, some have multiple eras, such as one for the reign of each leader.
249     * The exact meaning is determined by the {@code Chrono}.
250     * <p>
251     * All correctly implemented {@code Era} classes are singletons, thus it
252     * is valid code to write {@code date.getEra() == SomeChrono.ERA_NAME)}.
253     *
254     * @return the chronology specific era constant applicable at this date, not null
255     */
256    Era getEra();
257
258    //-----------------------------------------------------------------------
259    /**
260     * Checks if the year is a leap year, as defined by the calendar system.
261     * <p>
262     * A leap-year is a year of a longer length than normal.
263     * The exact meaning is determined by the chronology with the constraint that
264     * a leap-year must imply a year-length longer than a non leap-year.
265     * <p>
266     * The default implementation uses {@link Chronology#isLeapYear(long)}.
267     *
268     * @return true if this date is in a leap year, false otherwise
269     */
270    boolean isLeapYear();
271
272    /**
273     * Returns the length of the month represented by this date, as defined by the calendar system.
274     * <p>
275     * This returns the length of the month in days.
276     *
277     * @return the length of the month in days
278     */
279    int lengthOfMonth();
280
281    /**
282     * Returns the length of the year represented by this date, as defined by the calendar system.
283     * <p>
284     * This returns the length of the year in days.
285     * <p>
286     * The default implementation uses {@link #isLeapYear()} and returns 365 or 366.
287     *
288     * @return the length of the year in days
289     */
290    int lengthOfYear();
291
292    //-------------------------------------------------------------------------
293    // override for covariant return type
294    @Override
295    ChronoLocalDate<D> with(TemporalAdjuster adjuster);
296
297    @Override
298    ChronoLocalDate<D> with(TemporalField field, long newValue);
299
300    @Override
301    ChronoLocalDate<D> plus(TemporalAmount amount);
302
303    @Override
304    ChronoLocalDate<D> plus(long amountToAdd, TemporalUnit unit);
305
306    @Override
307    ChronoLocalDate<D> minus(TemporalAmount amount);
308
309    @Override
310    ChronoLocalDate<D> minus(long amountToSubtract, TemporalUnit unit);
311
312    //-----------------------------------------------------------------------
313    /**
314     * Calculates the period between this date and another date as a {@code Period}.
315     * <p>
316     * This calculates the period between two dates in terms of years, months and days.
317     * The start and end points are {@code this} and the specified date.
318     * The result will be negative if the end is before the start.
319     * <p>
320     * The calculation is performed using the the chronology of this date.
321     * If necessary, the input date will be converted to match.
322     * <p>
323     * The result of this method can be a negative period if the end is before the start.
324     * The negative sign will be the same in each of year, month and day.
325     * <p>
326     * This instance is immutable and unaffected by this method call.
327     *
328     * @param endDate  the end date, exclusive, which may be in any chronology, not null
329     * @return the period between this date and the end date, not null
330     * @throws DateTimeException if the period cannot be calculated
331     * @throws ArithmeticException if numeric overflow occurs
332     */
333    Period periodUntil(ChronoLocalDate<?> endDate);
334
335    //-----------------------------------------------------------------------
336    /**
337     * Combines this date with a time to create a {@code ChronoLocalDateTime}.
338     * <p>
339     * This returns a {@code ChronoLocalDateTime} formed from this date at the specified time.
340     * All possible combinations of date and time are valid.
341     *
342     * @param localTime  the local time to use, not null
343     * @return the local date-time formed from this date and the specified time, not null
344     */
345    ChronoLocalDateTime<D> atTime(LocalTime localTime);
346
347    //-----------------------------------------------------------------------
348    /**
349     * Converts this date to the Epoch Day.
350     * <p>
351     * The {@link ChronoField#EPOCH_DAY Epoch Day count} is a simple
352     * incrementing count of days where day 0 is 1970-01-01 (ISO).
353     * This definition is the same for all chronologies, enabling conversion.
354     *
355     * @return the Epoch Day equivalent to this date
356     */
357    long toEpochDay();
358
359    //-----------------------------------------------------------------------
360    /**
361     * Compares this date to another date, including the chronology.
362     * <p>
363     * The comparison is based first on the underlying time-line date, then
364     * on the chronology.
365     * It is "consistent with equals", as defined by {@link Comparable}.
366     * <p>
367     * For example, the following is the comparator order:
368     * <ol>
369     * <li>{@code 2012-12-03 (ISO)}</li>
370     * <li>{@code 2012-12-04 (ISO)}</li>
371     * <li>{@code 2555-12-04 (ThaiBuddhist)}</li>
372     * <li>{@code 2012-12-05 (ISO)}</li>
373     * </ol>
374     * Values #2 and #3 represent the same date on the time-line.
375     * When two values represent the same date, the chronology ID is compared to distinguish them.
376     * This step is needed to make the ordering "consistent with equals".
377     * <p>
378     * If all the date objects being compared are in the same chronology, then the
379     * additional chronology stage is not required and only the local date is used.
380     * To compare the dates of two {@code DateTimeAccessor} instances, including dates
381     * in two different chronologies, use {@link ChronoField#EPOCH_DAY} as a comparator.
382     *
383     * @param other  the other date to compare to, not null
384     * @return the comparator value, negative if less, positive if greater
385     */
386    @Override
387    int compareTo(ChronoLocalDate<?> other);
388
389    //-----------------------------------------------------------------------
390    /**
391     * Checks if this date is after the specified date ignoring the chronology.
392     * <p>
393     * This method differs from the comparison in {@link #compareTo} in that it
394     * only compares the underlying date and not the chronology.
395     * This allows dates in different calendar systems to be compared based
396     * on the time-line position.
397     * This is equivalent to using {@code date1.toEpochDay() &gt; date2.toEpochDay()}.
398     *
399     * @param other  the other date to compare to, not null
400     * @return true if this is after the specified date
401     */
402    boolean isAfter(ChronoLocalDate<?> other);
403
404    /**
405     * Checks if this date is before the specified date ignoring the chronology.
406     * <p>
407     * This method differs from the comparison in {@link #compareTo} in that it
408     * only compares the underlying date and not the chronology.
409     * This allows dates in different calendar systems to be compared based
410     * on the time-line position.
411     * This is equivalent to using {@code date1.toEpochDay() &lt; date2.toEpochDay()}.
412     *
413     * @param other  the other date to compare to, not null
414     * @return true if this is before the specified date
415     */
416    boolean isBefore(ChronoLocalDate<?> other);
417
418    /**
419     * Checks if this date is equal to the specified date ignoring the chronology.
420     * <p>
421     * This method differs from the comparison in {@link #compareTo} in that it
422     * only compares the underlying date and not the chronology.
423     * This allows dates in different calendar systems to be compared based
424     * on the time-line position.
425     * This is equivalent to using {@code date1.toEpochDay() == date2.toEpochDay()}.
426     *
427     * @param other  the other date to compare to, not null
428     * @return true if the underlying date is equal to the specified date
429     */
430    boolean isEqual(ChronoLocalDate<?> other);
431
432    //-----------------------------------------------------------------------
433    /**
434     * Checks if this date is equal to another date, including the chronology.
435     * <p>
436     * Compares this date with another ensuring that the date and chronology are the same.
437     * <p>
438     * To compare the dates of two {@code DateTimeAccessor} instances, including dates
439     * in two different chronologies, use {@link ChronoField#EPOCH_DAY} as a comparator.
440     *
441     * @param obj  the object to check, null returns false
442     * @return true if this is equal to the other date
443     */
444    @Override
445    boolean equals(Object obj);
446
447    /**
448     * A hash code for this date.
449     *
450     * @return a suitable hash code
451     */
452    @Override
453    int hashCode();
454
455    //-----------------------------------------------------------------------
456    /**
457     * Outputs this date as a {@code String}.
458     * <p>
459     * The output will include the full local date and the chronology ID.
460     *
461     * @return the formatted date, not null
462     */
463    @Override
464    String toString();
465
466    /**
467     * Outputs this date-time as a {@code String} using the formatter.
468     *
469     * @param formatter  the formatter to use, not null
470     * @return the formatted date-time string, not null
471     * @throws DateTimeException if an error occurs during printing
472     */
473    String toString(DateTimeFormatter formatter);
474
